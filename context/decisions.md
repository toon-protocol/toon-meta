# Architectural Decisions

Curated, durable decisions. ADR-lite: each is *decision → why*.

## Payment & protocol

- **Messages and money are one packet.** Every monetized flow = a single ILP PREPARE carrying both the TOON-encoded event and its payment. *Why:* the core protocol thesis; no separate invoice/settle round-trip.
- **Prepaid, supply-driven pricing.** Providers advertise price in a replaceable Nostr event (`SkillDescriptor`, kind:10035); the request packet's amount IS the payment. `settleCompute()` deprecated. *Why:* removes request-for-quote latency; the network can't distinguish rails.
- **TOON's value layer is the signed payment-channel claim — not SPSP / STREAM / payment-pointers.** SPSP kinds (23194/23195) were removed. *Why:* a write is one packet + one balance-proof claim; there's no stream to chunk, no quoting. *Transports:* claims ride over **BTP/WebSocket** (duplex sessions + inter-connector peering) **and ILP-over-HTTP** (`POST /ilp`, the one-shot edge/onboarding ingress, with an HTTP→BTP upgrade) — see `rfc-0023`, `rfc-0035`. *Atomicity:* multi-hop uses packet-level **execution-condition/fulfillment** (active when NIP-59 claim-wrapping supplies the preimage); there is **no on-chain HTLC escrow**. See the `rfc-*` skills for per-RFC detail.
- **USDC is the sole user-facing token.** *Why:* simplicity; operator staking tokens stay invisible to relay users.
- **Claims are per-chain balance proofs:** EIP-712 (EVM), Ed25519 (Solana), Pallas/Schnorr zk (Mina). Settlement is **in-process multi-chain** (not RFC-0038's separate service), redeeming via `claimFromChannel` at a per-peer threshold.
- **Payment proxy (TOON in front of any HTTP service).** A connector at the edge can act as a **payment proxy server** — proxying the payment the way nginx fronts TLS: onboard via **x402** (HTTP 402), pay one-shot via **transparent HTTP-in-ILP** (the raw HTTP request rides in the opaque ILP `data`), upgrade to **BTP** for sessions; the backend stays payment-oblivious and the connector never parses `data`. *Why:* agent-first adoption — ride x402's installed base, and channels + n-hop routing beat per-request on-chain settlement at volume. **Path A status: core shipped on connector `main`** — proxy handler, x402 greeting, `h402Fetch` shim, RFC 9421 binding, and `RouteTermination` config all exist on `main` and were verified by a real paid round-trip (proven live at `connector.pay.toonprotocol.dev`, reusable artifact `connector/deploy/pay-edge/`). Also shipped: the devnet multi-chain roundtrip harness + connector naming + Porkbun DNS (connector PR #245, merged) and the `deploy/pay-edge/` deploy bundle (connector PR #252, merged; supersedes closed connector PR #246). Only future item: transparent cross-chain FX. See [`payment-proxy.md`](../docs/payment-proxy.md).

## Boundaries

- **Claim validation lives ONLY in the connector.** `core` never imports the connector (structural `EmbeddableConnectorLike` interface); `sdk` dynamically imports it only to auto-create one; the `payment-handler-bridge` dispatches an *already-paid* packet to business logic. *Why:* the connector is the only component holding channel state — re-validating downstream is double work and incorrect.
- **Apex / free-forward.** Operators run an apex (the connector as a proxy-server layer, `g.toon`) + child nodes; parent→child packets carry no per-packet claim (settled in aggregate). Children must be `relation:'child'` and tag `g.toon` as parent. *Why:* one paid hop at the edge; children earn via aggregate settlement.
- **Trust degrades; money doesn't.** (TEE) Attestation state changes never trigger payment-channel closure. *Why:* trust is a gradient, not a gate.

## Repo split (2026-06)

- **Polyrepo with npm + pinned-digest coupling** (not a monorepo, not submodules). *Why:* per-team ownership; teams build/test/release without rebuilding the world. Mirrors how the connector was already consumed.
- **`toon` (core+sdk) is libraries only; connector is an optional peer.** *Why:* the library layer must build/publish independent of the payment engine.
- **The connector owns & publishes `@toon-protocol/mina-zkapp`.** *Why:* one canonical Mina channel contract; the connector already depends on it, and it was unpublished/`private`, breaking installs.
- **Publish via `pnpm publish` / changesets, never `npm publish`.** *Why:* `npm publish` shipped unresolved `workspace:*`, making `sdk@0.5.0`/`town@0.4.0` uninstallable.
- **`g.toon` is the canonical apex wire nodeId.** *Why:* it's baked into the connector + child parent tags and every party must agree on it, or paid forwarding breaks (T00/F06) — so it's a load-bearing on-wire term, not cosmetic. **Status:** the live devnet and the epic-44 docs use **`g.toon`** (children `g.toon.<type>`, env prefix `PROXY_*`; live ILP edges e.g. `connector.pay.toonprotocol.dev/ilp`, `proxy.store.devnet.toonprotocol.dev/ilp`). "Connector" remains the repo/product name — only the on-wire nodeId + env prefix are the `g.toon` axis. A cleanup to purge remaining legacy `g.connector` references in favor of `g.toon` is a **pending follow-up**.

## Knowledge architecture

- **toon-meta is the shared context + skills hub**, distributed as a Claude Code plugin; each repo's `CLAUDE.md` links here. BMAD framework skills and the raw planning dump were removed in favor of this curated `context/`. *Why:* repos need accurate, low-noise shared context, not a 3000-line planning archive.

## Software factory (Forge)

- **Forge is a factory *consumer*, not just the factory *producer*.** Forge (`toon-protocol/Forge`, the factory manager from [#198](https://github.com/toon-protocol/toon-meta/issues/198)) runs its own `.sandcastle/` factory like every other execution repo — it is not exempt because it builds factories. *Why:* org doctrine is that every execution repo runs the two-zone pipeline; a factory manager that isn't itself manufactured has no dogfood.
- **Forge bootstraps hand-rolled, then self-stamps (the self-hosting-compiler order).** Forge's *first* `.sandcastle/` is hand-applied from the relay-extracted recipe (a stage-0 gate, exactly like the #178 repos); `forge-core`/`forge-cli` are then developed *under* that gate; once the CLI works, `forge new` regenerates Forge's own `.sandcastle/`. *Why:* it is the only bootstrap order that never runs **ungated codegen to produce the gate that gates it**; and it makes Forge's hand-rolled factory an instance of the very templates Forge stamps, so hand-rolling Forge = authoring those templates. *Rejected:* self-stamp-from-source-day-one (backwards — ungated codegen); gate-only-until-self-host (leaves Forge un-factoried during the whole build).
- **Two distinct self-hosting proofs, self-host first — they are non-redundant.** (1) *Self-host:* `forge new` regenerates Forge's own `.sandcastle/` and reaches **green parity** with the hand-rolled bootstrap — proves the CLI runs end-to-end and the templates are complete. (2) *External re-stamp:* `forge new` re-stamps an existing #178 repo (relay) to green parity with **its** hand-rolled `.sandcastle/` — proves Forge generalizes to a factory it did not author. *Why keep both:* self-host is incestuous (same authors wrote Forge's hand-rolled factory and the templates), so it cannot catch generalization bugs; the external re-stamp is the load-bearing surprise. Making self-host *replace* the external re-stamp would let Forge pass acceptance having only ever stamped itself — the exact case where "works on the repo it was tuned against" hides the bug.
- **Forge self-stamps with `forge new --blank` + the `node-pnpm` environment — it mints no archetype.** *Why:* an archetype is *named for the domain you build* and *only exists after a pilot proves it*; Forge builds tooling, not a domain, and there is exactly one Forge, so it can't earn an archetype (a `forge`/`factory` archetype would be a category error). *Deferred, not decided here:* the 6 boring node-pnpm repos share the same gap — no `game`-like archetype fits them — so "what do plain node repos pass to `forge new`" (a possible base `node-lib` archetype) rides as a **separate** question; Forge's `--blank` node-pnpm factory is the natural extraction source if the org ever wants one, exactly as relay was for the templates.
- **Forge has a *double* oracle-protected zone — its product IS oracle machinery.** Unlike other repos (one protected zone = their own `verify/`), the `templates/**` gate runners + archetype oracle skeletons Forge ships **are oracle code for downstream repos**. So two zones are protected under Rule 4 (diff-path check + CODEOWNERS): (i) Forge's own `verify/` + gate scripts, and (ii) the shipped `templates/**` gate runners. No PR touches a template gate-runner and the `forge-core` code that emits it in the same diff. *Why:* otherwise an agent PR could "green" a broken stamper by rewriting the very machinery that judges it.
- **Golden-stamp fixtures are the Forge oracle, and their regeneration is dispatch-gated.** Forge's signature gate tier (T2) diffs `forge new` output against **committed golden fixtures** of the stamped `.sandcastle/`/workflows/Dockerfile/`factory.toml`. Regenerating those fixtures runs **only** through `golden-regen.yml` behind the `oracle-owners` environment (Rule 5). *Why:* if one PR could edit both a template and the golden fixture that pins its expected output, the golden test would self-certify — the agent regenerates the expectation to match whatever it changed, and the check becomes theater.
- **Forge's steady-state factory runs `forge-core`; raw `@ai-hero/sandcastle` is only the throwaway stage-0 scaffold.** The hand-rolled bootstrap runs raw sandcastle `main.ts` (forge-core doesn't exist yet, identical to the 8 repos); the self-stamp **swaps the engine** to `forge-core` + `factory.toml`, and that forge-core-driven factory is what Forge runs from then on. *Why:* a factory manager whose own factory doesn't run its own engine isn't eating its dogfood, and it would leave forge-core's orchestration path with no repo exercising it in anger.
- **"Green parity" is behavioral, never byte-identical.** Because raw-sandcastle `.sandcastle/` and forge-core `.sandcastle/` are structurally different by construction, both self-host and external-re-stamp parity mean *"the gate goes green and a real `agent:implement` PR merges end-to-end under the forge-core-driven factory"* — **not** a file diff against the hand-rolled scaffold. *Why:* the self-stamp swaps the engine shape, so a textual diff would always fail and measures the wrong thing.
- **Forge is the 9th registry row, hand-added at bootstrap — not withheld for self-registration.** Forge joins the going-forward set (8→9) with its `FACTORY.md` row added by hand when its hand-rolled factory is scaffolded, and is marked the only `forge-core`-driven row. The self-stamp's `forge new` registration PR is then an **idempotent re-registration** (already-registered → parity/no-op). *Why:* withholding the row until self-registration would leave a *genuinely live* factory absent from the org's source of truth — "unregistered = doesn't exist" is a validation rule, not reality, and a running-but-unregistered Forge makes `FACTORY.md` lie by omission. The "both directions" guard (unregistered → doesn't exist; pin mismatch → fails) is proved with throwaway fixture registries + the external re-stamp, so registry accuracy needn't be sacrificed to test it.
- **Self-host parity is a nightly/dispatch tier, not a per-PR gate.** T4 (a forge-core-driven self-stamp reaching green/behavioral parity; `forge doctor` green on HEAD) runs nightly + on dispatch. *Why:* a full self-stamp is expensive and cross-referential; gating every PR on it would make Forge un-iterable, and regressions are better surfaced as triaged nightly issues than as a per-PR wall.
- **Forge inherits the factory runtime policy ([#202](https://github.com/toon-protocol/toon-meta/issues/202)), and its templates become that policy's canonical home.** #202 makes per-role model tiering (planner = `claude-opus-4-8`; implementer + reviewer = `claude-sonnet-5`) and the ~60% context ceiling (small slices + pre-dispatch splitting + stop-and-handoff prompts) org-wide. Two consequences for Forge: (i) **`FACTORY_SPEC.md` must express per-role models** — forge-core wraps sandcastle (whose four roles all default to Opus), so `factory.toml` needs a per-role model field (in `[loop]` or a `[models]` block) for forge-core to configure them; this is a spec requirement the epic's original `[loop]`/`[[oracle.tier]]` sketch omitted. (ii) **Forge's protected `templates/**` ship #202's tiering + 60%-handoff prompt as baked-in defaults**, so the policy propagates by `forge upgrade` (one template change + N gated PRs) instead of an 8-repo hand sweep — #202 is the poster-child example of why Forge exists. *Sequencing:* #202's policy must land in the hand-rolled 8 (the relay-proven pattern) **before** forge-core extracts templates, else Forge extracts a pre-#202 pattern and needs an immediate `forge upgrade`; so #202 is a dependency-edge *into* forge-core extraction, not a parallel afterthought. Forge's own two stories adopt the tiering at creation (stage-0 sets impl+review = Sonnet-5 by hand; self-host inherits it via forge-core).
- **Forge's factory ships as two stories at opposite ends of the epic, not one.** *Early* — `forge:bootstrap`: hand-rolled stage-0 `.sandcastle/` (raw sandcastle, relay recipe) + hand-added `FACTORY.md` row; depends only on the repo scaffold + relay recipe (**not** `FACTORY_SPEC.md`), and **blocks** forge-core (forge-core is developed under it). *Late* — `forge:self-host`: swap the engine to forge-core-driven `factory.toml`, reach green parity (the self-host checkpoint), wire T2 golden-stamp + T4 nightly self-parity + the double protected zone; **blocked by** forge-cli + templates, sequenced alongside/just after the external relay re-stamp, before the archetype catalog. *Why:* keeps "Forge is itself a factory" true from the earliest possible moment (a plain gate, day one) while deferring the expensive forge-core dogfood to when the CLI can deliver it — so Forge is never building factories while lacking one, and its longest/riskiest build window (forge-core + CLI) is gated.
