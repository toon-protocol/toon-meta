## What Went Well

### The Live E2E Gate Found What Code Review Could Not — Again

Story 47.5 surfaced **two product-level bugs invisible to 994+ green unit tests**:

- **BUG-1 (D2):** `ConnectorConfigGenerator.toYaml()` omitted `chainProviders` → connector `/admin/earnings.json` returned 503 → entire earnings plane silently degraded. Unit tests had no `chainProviders` field to check because the spec didn't require one until the gate exercised it live. Fixed in this branch (schema + defaults + generator + hs-writer + validator + 12 new tests).
- **BUG-2 (D1):** Connector v3.6.2's `ClaimReceiver` was constructed but never stored as `this._claimReceiver`; `AdminServer` got constructed without it; earnings always 503'd. Fixed upstream (connector PR #72, v3.6.3). Pre-existing latent defect — exposed only when a real townhouse-API consumer tried to read earnings.

Epic 46's bet on A4 (live gate as structural per-epic ritual) compounds: Epic 47 is the second consecutive epic where the gate paid for itself in week one.

### Cross-Repo Coordination Tightened to a Single Day

Finding Q in Epic 46 took multiple iterations across connector + townhouse repos. BUG-2 in Epic 47 was: gate found it 2026-05-13 → connector PR #72 cut, reviewed, merged → connector v3.6.3 published via Release workflow #25813311615 → townhouse-api built against v3.6.3 → gate run 3 PASS — **all in one day**. The shape established in Epic 46 (fix upstream, version-bump locally) is now muscle memory.

### Story 47.2's Wire-Shape Surgery Was the Right Time

Adding `status: 'ok' | 'connector_unavailable'` to the wire shape in 47.2 (D1 resolution) rather than bolting it onto 47.4 meant 47.3, 47.4, and 47.5 all built on the operator-visibility contract. The downstream gate then asserted `status === 'ok'` as a precondition for every peer-bucket check — turning what could have been silent zero-fill into a load-bearing signal.

### Snapshot Writer's Negative-Baseline Fix Caught a Real Class of Bug

P11 in 47.3 — `subOrZero` clamps negative baseline to `'0'` — was an Edge Case Hunter find. Without it, a corrupt snapshot row with a negative `claimsReceivedTotal` would have inflated the apparent delta, showing operators fictional earnings. The fix is one-line; the test asserting it isn't. Worth keeping the pattern: clamp at the consumer boundary, not the producer.

## Challenges

### 1. Code-Review Patch Volume Doubled (33 → 76)

Epic 46 totalled ~33 patches across 4 stories. Epic 47 hit 76 across 5 stories. Per-story patch counts: 9, 23, 15, 7, 22. Two stories crossed 20 patches:

- **47.2 (23 patches):** D1 wire-shape addition (`status` field) cascaded to 4 patches across aggregator + tests + e2e shape guard. Many other patches were "tighten before downstream stories build on this" — appropriate for a foundational surgery story.
- **47.5 (22 patches):** Pure test/integration patches — the gate test surface (Ajv strict, status-narrowing, `peers[]` guards, regex tightening, env save/restore, `AbortSignal.timeout`, Docker filter anchoring, etc.). Reviewer was right to ask; story author wrote a working gate that the patches hardened into a *robust* gate.

**Read:** patch volume is a quality metric, not a defect signal. The 76 patches are all in the codebase + tests, not in `deferred-work.md`. Compare with 47.3's discipline: D1 deleted the PIPE_BUF split branch as actually-not-needed, P_D1 dismissed the spy test that no longer had a branch to spy on. Reviewer made the right call; author corrected immediately.

### 2. Two Product-Level Bugs Reached the Gate

BUG-1 (chainProviders) and BUG-2 (ClaimReceiver not wired) both lived in product code that 994 unit tests called green. Both required a real connector + a real townhouse-api in the same Docker network to surface. Neither was a regression of an Epic-46-retro action item — they're fundamentally new shapes:

- **BUG-1 (compose-config seam):** The `TownhouseConfig` schema didn't model `chainProviders`; the generator didn't emit them; the connector requires them. The contract between Town's config and the connector's `AccountManager` was implicit. *A4' (gate dry-run) might have surfaced this earlier in dev*.
- **BUG-2 (cross-repo latent defect):** Connector v3.6.2 (Epic 46 shipped this) had unused-instance-field code that no consumer exercised until earnings.

Worth asking: do we want a "connector × townhouse-api smoke test" that runs against every connector publish without waiting for an epic gate?

### 3. Gate Iteration Cost — 3 Runs, Same as Epic 46

Run 1: 2 bugs found. Run 2: 5/5 with chainProviders YAML workaround. Run 3: clean 5/5 in 60.79s after product fix + connector v3.6.3.

Iteration was *faster* than Epic 46 (single-day cross-repo turnaround), but still 3 runs. The compose-template contract test (A3', not yet landed) wouldn't have caught chainProviders directly, but the *pattern* — a structural assertion that every connector-required field is emitted by the generator — would. Worth widening A3' from "compose vars" to "any config field the connector requires."

### 4. Real-Claim Driving Is Genuinely Hard (D3)

The investigation Explore-agent report on D3 is worth re-reading: `BtpRuntimeClient.sendClaimMessage()` is reachable from host, the auth token isn't a blocker, but the apex's `ClaimReceiver.verifyClaim()` requires a real EIP-712 signature over a balance proof from a real payment channel — and the signer-key lives inside the town container by design while chainProviders point at a dead RPC.

To unblock B.3.a/B.3.b, we need real chain machinery in the gate (Anvil + deployed channel registry from `scripts/sdk-e2e-infra.sh`, repointed `chainProviders.rpcUrl`). That's ~150 LOC of cross-network plumbing — exactly what Epic 50 is for. PM-deferring this was correct.

**But this means** Epic 47's `peers[].type === 'external'` AC (#1 step 6) and `peers[].byAsset[].today/month/year/lifetime` non-zero assertions are only *probably* correct — they were exercised in gate run 3 because the connector happened to surface a zero-claim external peer, but no real claim flow was driven end-to-end. The contract canary + unit tests give us shape confidence; full live confidence waits for Epic 50.

## Key Insights

1. **The gate as second-stage filter is now structural.** Epic 45 introduced it (A4 of Epic 45's retro). Epic 46 proved it (14 findings). Epic 47 reinforces it (BUG-1 + BUG-2). It is no longer up for debate. Every future epic in this series ships a `*-live-e2e-gate-*` story.

2. **Code review's adversarial layer is calibrated correctly.** 5 reviewers found a few real defects (P11 negative baseline, BUG-2 captured-reference race) and many *hardening* opportunities (Ajv strict, regex tightening, env save/restore). Both classes are valuable; conflating them is what makes code review feel like noise. The dated `### Review Findings` block with `[Review][Decision] / [Patch] / [Defer] / [Dismiss]` triage is the right shape — keep it.

3. **D-blockers separate "tested" from "shipped."** The five-decision pattern in 47.5 (D1–D5) blocked flip-to-done on real-world artifacts (PR merge SHAs, published version digests, env-var escape hatches), not paper sign-offs. This is the muscle Epic 47 built on top of Epic 46's gate; it's how the gate produces actionable closure instead of perpetual "almost-done" state.

4. **Cross-repo turnaround is now a day, not a week.** Epic 46's connector v3.6.2 (Finding Q) took multiple iterations. Epic 47's connector v3.6.3 (BUG-2) was same-day: investigate → PR → review → merge → publish → gate rerun. The compounding effect Epic 46's retro predicted is now load-bearing.

5. **Action-item follow-through dipped (5/5 → 2/4 + 2 partial).** The A3' compose-env-contract test is the most concrete miss — sub-minute work that didn't land. Without it, the *next* epic that touches compose templates pays the price. Worth scheduling explicitly into Epic 48 or its successor rather than carrying it forward as a vague debt.
