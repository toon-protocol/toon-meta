## DEEP STORY ANALYSIS — Patterns Across Epic 11

Bob (Scrum Master): "I reviewed all 18 story records. Here are the patterns."

### Common Strengths

**1. Acceptance Criteria depth and specificity — 18/18 stories**
Every story entered development with numbered, testable ACs. Dev agents cited ACs explicitly in completion notes (e.g., "AC-6 verified — 64-char hex hash confirmed"). This made code reviews efficient: reviewers could immediately cross-reference claims against criteria.

**2. Multi-pass code review discipline — 114 issues caught, 106 fixed**
The 3-pass review pattern (functional → security → adversarial) consistently surfaced issues that a single pass would have missed. Story 11-1 exemplified this: 24 issues across 3 passes, zero remaining after fixes. The critical `Cargo.lock` finding in pass 1 would have caused reproducibility failures in production.

**3. Key architectural decisions documented inline**
Dev Notes sections captured non-obvious decisions with rationale. Examples: the ESM/CJS bridge for napi-rs (11-1), TOC file-read approach for `hash()` because `Memvid.toc` is `pub(crate)` (11-1), why `RNG.setSeed()` must be called before each dungeon run (11-15). This pattern saved future agents from rediscovering the same constraints.

**4. Graceful browser/server boundary enforcement (11-9)**
Story 11-9's AC-3 explicitly prohibited Node.js `Buffer` in favor of browser-safe `atob()`. The dev agent caught this and implemented correctly on first pass. Client package remained browser-safe — zero regression to ditto React SPA compatibility.

**5. Golden test vectors as first-class artifacts (11-2, 11-4)**
The 26 golden vectors in `packages/pet-circuit/test-vectors/golden-vectors.json` serve as both test fixture and living specification. When circuit logic and game engine diverge, the golden vectors are the ground truth. This pattern should be applied to all ZK story pairs going forward.

### Common Struggles

**1. napi-rs path resolution was more complex than planned (11-1)**
The story spec assumed `../../memvid` path for the Cargo dependency (2 levels up). The actual path was 3 levels: `packages/memvid-node/` → `packages/` → `TOON-Protocol/` → `Documents/` → `memvid/`. This type of "spec assumes flat layout" mismatch will recur whenever external sibling repos are dependencies.

**2. o1js WASM incompatibility with vitest (11-2, 11-3)**
The vitest root config was extended to exclude `pet-circuit` (uses Jest, WASM incompatible). This was resolved in the epic start baseline, but it means o1js tests run in Jest while everything else runs in vitest — two test runner configs to maintain. Story 11-2 and 11-3 tests marked `@slow` due to circuit compilation time (2-5 minutes per full proof run).

**3. rot.js global RNG singleton creates threading footgun (11-15, 11-17)**
`RNG.setSeed()` resets a global singleton. In single-threaded Node.js, sequential test execution (`--runInBand`) is safe. But this is an invisible footgun for anyone who tries to parallelize dungeon DVM handlers via Worker threads. The constraint is documented but requires active enforcement — any future parallelism work will hit this.

**4. Static exchange rate oracle (11-11)**
Story 11-11 (Cross-Chain DVM Pricing) implemented a hardcoded 1 MINA = 0.50 USDC rate with a documented placeholder. This is flagged in 3 NFR assessments as a concern. The architecture is sound — the oracle interface is in place — but the static rate means real-world usage requires a live price feed before Epic 12/13 can safely depend on multi-token pricing.

**5. Proof queue WAL persistence not addressed (11-5, 11-7)**
The optimistic proof pipeline (DVM returns optimistic result, ZK proof generated async, settlement deferred) stores pending proofs in-memory. If the DVM process restarts, unproven interactions are lost. This was flagged as a known limitation in story 11-5 dev notes and was documented but not resolved within Epic 11 scope.

### Breakthrough Moments

**1. BLAKE3 composite hash design (11-1)**
Excluding the HNSW vec index from the composite hash was the key insight. Vec indices are non-deterministic (graph construction order varies). The 5-segment deterministic hash (frames, lex, time, temporal, sketch) enables cross-platform reproducibility — essential for ZK circuit integration. The 100-iteration determinism property test is the quality gate that proves it.

**2. Three-tier trust model execution (11-2/11-4/11-5)**
Full ZK → verifiable content → DVM-attested tiers shipped without compromise. Story 11-7's E2E test proved the full pipeline: optimistic result in milliseconds, ZK proof generated async, settlement via Mina. The architecture held up under implementation.

**3. Dungeon extension velocity (11-15 through 11-18)**
Four stories planned and delivered in Sprint 5, added mid-epic. The `DungeonGameEngine` was clean enough that Story 11-16 (stat bridge), 11-17 (DVM handler), and 11-18 (adventure log) composed on top without architectural surgery. Extensibility of the pet-dvm package pattern proved out.

**4. Browser/server boundary enforcement via AC design (11-9)**
Making "no Node.js Buffer" an explicit AC (rather than hoping the dev would infer it) meant the browser compatibility constraint was enforced by the story spec, not discovered in PR review. This is a lesson in where constraints belong.

## WHAT DIDN'T GO WELL / CHALLENGES

Bob (Scrum Master): "Now the challenges — no blame, systems focus."

1. **Two test runners (Jest + vitest) create split maintenance burden.** o1js WASM incompatibility with vitest is a framework constraint, not a team mistake. But the split config (Jest for pet-circuit, vitest everywhere else) is ongoing friction. Every new story touching pet-circuit must remember the Jest exception.

2. **Proof queue WAL persistence deferred without a resolution plan.** In-memory proof queue is a known limitation that could cause data loss on DVM restart. It was documented but not scheduled. This should become a backlog story before Epic 12's settlement patterns depend on it.

3. **Static exchange rate oracle is a time bomb.** Story 11-11's hardcoded 1 MINA = 0.50 USDC rate is acceptable for MVP but must be replaced before real usage. The interface is ready; the live feed implementation is not. Three NFR assessments called this out — it deserves a dedicated story or at minimum a P0 AC in whichever Epic 12 story first touches multi-token pricing.

4. **rot.js global RNG singleton is invisible to new contributors.** The `--runInBand` requirement for pet-dvm tests is documented, but the *reason* (global RNG state) isn't obvious from test config alone. Anyone parallelizing tests in the future will encounter hard-to-debug non-determinism. A code comment in `DungeonGameEngine.ts` at the `RNG.setSeed()` call would make this visible.

5. **napi-rs binary not yet in Docker image.** The `memvid-node` native addon requires Rust build toolchain at build time. The Dockerfile doesn't currently include it. The Docker path was noted as a future concern in 11-1 dev notes, but it's now a real blocker for any deployment pipeline that containerizes pet-dvm.

## KEY INSIGHTS AND LESSONS LEARNED

Bob (Scrum Master): "The big takeaways from Epic 11."

1. **ZK circuit + game engine story pairing requires golden vectors as a mandatory deliverable.** When two systems must produce identical outputs (circuit and engine), shared test fixtures are not optional. The `golden-vectors.json` pattern from 11-2/11-4 should be a required AC for all future ZK story pairs.

2. **AC explicitness is the primary bug prevention mechanism.** "No Node.js Buffer" as an explicit AC (11-9) caught a browser/server boundary issue before it became a runtime bug. Browser compatibility constraints, threading constraints, platform constraints — all should be in the ACs, not in tribal knowledge.

3. **Multi-pass code review with adversarial pass is worth the investment.** 114 issues found across 18 stories. The Cargo.lock critical finding in 11-1 would have caused reproducibility failures in production. The MAX_SAFE_INTEGER and unpinned SHA patterns recurred enough that they should be in the story template.

4. **New package introduction should include Docker/deployment path as a P0 story task.** Story 11-1 shipped a functional napi-rs package without a Docker deployment plan. The binary distribution problem (Rust toolchain in Docker) is now deferred debt. Future package stories should include "CI/CD path verified" as an AC.

5. **Scope expansion mid-epic is viable when the base architecture is clean.** Sprint 5 (4 stories added 2026-04-08) was delivered successfully because stories 11-4/11-5 had clean engine/handler boundaries. Architectural discipline in early stories directly enabled late-epic agility.

6. **Party Mode research sessions are high-value inputs.** Both the original epic design (Party Mode 2026-04-05/06) and the dungeon extension (Party Mode 2026-04-08) produced decision-quality outputs in a single session. The research artifact quality is consistent with production planning.
