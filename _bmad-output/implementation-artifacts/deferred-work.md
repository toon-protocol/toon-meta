- Likely candidates: (a) relayUrls not plumbed through to `startMill` despite JSON+env both set; (b) cross-container reachability — does the town relay's `NostrRelayServer` bind `0.0.0.0:7100` (reachable from the Mill container) vs `127.0.0.1`? **Harness fixes already landed this session (correct, but insufficient alone)** in `townhouse-dvm-arweave-e2e.test.ts` + `townhouse-e2e-local-hs.sh`: P7 connector pin 3.7.0→3.7.1; P8 `MILL_MNEMONIC` env (Mill rejects bare secretKey); P9 kind:10032 author filter → Mill's real mnemonic-derived pubkey captured from `mill_ready`; P10 `MILL_RELAYS`/`TOON_CONNECTOR_URL`/`TOON_PARENT_PEER_ID` env wiring; DN3 AC #3 DSEQ-agnostic. ### Resolution (Story 50.4, 2026-05-30) — AC #1 root cause + fix

**AC #1 — actual root cause (deeper than the candidate list).** The three candidates were each verified; the *dominant* blocker was a fourth, architectural one not on the list:

1. **Candidate (a) `relayUrls` empty — RULED OUT.** `startMill()` hard-throws `MillConfig.relayUrls MUST be a non-empty array` (`mill.ts:420`). Mill reached `mill_ready` with `swapPairCount:1`, so `relayUrls` was populated; the publish *was* attempted.
2. **Candidate (c) swallowed logger — REAL but secondary.** `startMill()`'s default logger is `noopLogger()` (`mill.ts:494`; `entrypoint-mill.ts` never passed a logger), and the publish was triple-swallowed: `void` fire-and-forget + `Promise.allSettled` + `logger.warn?.` no-op. Explains the *silence*, not the *non-delivery*.
3. **DOMINANT BLOCKER (d) — pay-to-write relay rejects the Nostr publish.** Mill advertised via `SimplePool.publish()` — a plain **unpaid Nostr WS `EVENT`**. The TOON relay rejects every unpaid WS write: `ConnectionHandler.handleEvent()` replies `OK=false, 'restricted: writes require ILP payment'` with **no kind:10032 exception and no self-write bypass**. kind:10032 only enters a relay's EventStore via (i) the relay self-writing its own (`town.ts:1180 eventStore.store`), or (ii) a peer's event arriving in an **ILP PREPARE** decoded at the node's packet handler → `createEventStorageHandler` → `eventStore.store` (`town.ts` handlePacket + `event-storage-handler.ts:55`). Mill did neither. So the advertisement was rejected on arrival regardless of (a)/(c).
4. **Secondary defect found while fixing (d): the embedded connector was never started.** In the `connectorUrl` ("embedded-with-parent") path, Mill constructed a `ConnectorNode` but **never called `.start()`** — so no BTP server, no parent dial, and `sendPacket()` had no live session. (Confirms 50.2's harness was never exercised: even the swap-handler inbound path was orphaned.)

**Fix (production path — Model 2 / the chosen option).**
- `startMill()` now **starts** the auto-created connector (`await autoCreatedConnector.start()`, guarded by `ownsConnector`; operator-supplied connectors remain the caller's responsibility — mirrors `startTown()`). Start resolves promptly even when the parent is unreachable (background BTP retry), so a flaky parent never aborts boot (R-8N2).
- New `MillConfig.peerInfoIlpDestination` (+ `peerInfoPricePerByte`) selects an **ILP advertisement** path: the TOON-encoded kind:10032 is routed to that ILP address via `createDirectIlpClient(connector).sendIlpPacket(...)` (mirrors `startTown()`'s self-advertise), with a bounded retry window (12×2s) because the parent BTP session establishes asynchronously after `start()`. When unset, falls back to the legacy SimplePool WS publish (vanilla Nostr relays only).
- **AC #2 (surface the failure):** the default no-op logger no longer hides failures — `docker/src/entrypoint-mill.ts` passes a real `MillLogger` (warn/error → stderr); the WS path logs `mill.peerInfo.ws_publish_unverified`; the ILP path logs a loud `error` (`mill.peerInfo.publish_failed` with destination + reason + hint) after exhausting retries instead of timing out 30s downstream.
- **Harness wiring (AC #3/#4):** `townhouse-dvm-arweave-e2e.test.ts` now sets `TOON_PEERINFO_ILP_ADDRESS=g.townhouse` + `TOON_PEERINFO_PRICE_PER_BYTE=0` on the Mill container, so Mill advertises through the apex connector → apex stores the kind:10032 → the test's `REQ {kinds:[10032], authors:[millPubkey]}` resolves. (Apex HS pricing is `FEE_PER_EVENT=0`, so amount `0` is accepted.)

**Verification (dev env):** `pnpm --filter @toon-protocol/mill build` clean; full mill unit suite **178 pass / 1 skip / 0 fail** (incl. 4 new Story 50.4 tests: ILP-route, ILP-priority, loud-fail-logs-error, WS-fallback); `docker/` `tsc --noEmit` adds **zero** new errors — the only `entrypoint-mill.ts` tsc errors (3, in the pre-existing `rateProvider` markup block, lines ~348/351) are present identically on the `2cf7626` baseline (confirmed by stashing the 50.4 change and re-running tsc), as are the 18 unrelated entrypoint-dvm/sdk/town errors; `entrypoint-mill.test.ts` 12/12 pass; eslint **0 errors** on changed source (42 pre-existing `no-non-null-assertion` warnings = house style). Standalone dist probe (`packages/mill/dist`) confirmed the ILP path end-to-end: `via:'ilp'` → `createDirectIlpClient(connector).sendIlpPacket({destination:'g.townhouse', amount:'0'})` fulfilled → `mill.peerInfo.published` on attempt 1. Mill image rebuilt from `docker/Dockerfile.mill` (id `sha256:99e96cf8…` — dev-local build; the published `ghcr.io/toon-protocol/mill:latest` digest is confirmed by the operator during the post-merge rebuild ritual; reconciled to match the epics `§ Status` note + `sprint-status.yaml` during the 50.4 code review, 2026-05-30), tagged `toon:mill-50-4` + `ghcr.io/toon-protocol/mill:latest` (the gate's `ensure_image_manifest` self-heals the manifest from this local digest); the rebuilt bundle contains the fix (`peerInfo.published`/`connector.started` markers present). **AC #4 — the full live Docker+Akash gate run (past the `beforeAll` kind:10032 wait into streamSwap settlement) is the operator post-merge ritual:** it requires live Akash EVM+Solana devnets redeployed via the operator's Console API (`leases.json` `anvil`/`solana` URLs; same external dependency as 50.3's 2026-05-30 run) plus the `.anyone` transport, so it is deferred to the operator exactly as 50.0/50.2/50.3 were. ### Live verification — /bmad-code-review live gate run (2026-05-31)

The code review **ran the AC #4 gate live** (Docker 29.5.2 + the live Akash anvil/solana devnets from `leases.json`) and **verified the advertisement blocker is resolved end-to-end**. The Mill-side fix worked as designed (`mill.connector.started` → `mill.peerInfo.built via:"ilp"` → `mill.peerInfo.published attempt 1`). Reaching green required **three additional harness fixes** in `townhouse-dvm-arweave-e2e.test.ts` that the live run exposed — none in Mill's runtime:

1. **Advertisement destination `g.townhouse` → `g.townhouse.town`.** The connector `localDelivery`-routes `g.townhouse` to the **DVM** HTTP handler (`:3300`, a `createNode()` registering only kind:5094), which rejects the kind:10032 with `F00: No handler registered for kind 10032`. The town relay's event store (with the `onDefault` `createEventStorageHandler`) is registered at `g.townhouse.town`. After the change, the apex FULFILLs and `[ConnectionHandler] Query returned 1 events` confirms the event is stored.
2. **TOON-aware read instead of nostr-tools.** The kind:10032 subscription used `SimplePool` (standard Nostr/JSON), but a TOON relay serializes every WS `EVENT` as a **TOON string** (`ConnectionHandler.ts:174 encodeEventToToonString`), so nostr-tools silently dropped every frame (`SimplePool.get` returned null even with `verifyEvent` disabled; a raw socket received the TOON frame). Replaced with a raw `ws` reader that `decodeEventFromToon`s the payload, polling the one-shot REQ until the ILP-path-stored event commits (race-free).
3. **Pubkey-scrape anchored to the `mill_ready` line** (code-review patch P3) so the kind:10032 author filter can't latch onto an earlier `"pubkey"` log field.

With these, `beforeAll` **clears** — execution proceeds past the kind:10032 wait into the EVM settlement (`openChannel`/`allowance`), which is **AC #4's bar met** ("advertisement no longer blocks the gate"). The run then fails at `ContractFunctionExecutionError: ECONNREFUSED 127.0.0.1:18545` — **no local EVM RPC**; the test expects a local anvil@18545 (`sdk-e2e-infra.sh up`, per the test-file header) which was not brought up in this direct `vitest` invocation. That EVM-chain prerequisite + the settlement assertions themselves are **Story 50.3's gate**, not 50.4's.
